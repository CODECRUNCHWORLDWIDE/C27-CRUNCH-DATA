# Week 1 — The Data-Engineering Landscape and Dimensional Modeling: Who Owns What, OLTP vs OLAP, the Warehouse / Lake / Lakehouse Lineage, Kimball Grain, Star vs Snowflake Schemas, Slowly-Changing Dimensions, and Surrogate Keys in Postgres

Welcome to **C27 · Crunch Data**, Week 1. This is the first week of the track, so we begin where every working data engineer begins — not with a distributed system, not with a streaming engine, but with a single Postgres database on a laptop and the most consequential decision in the entire discipline: **what does one row in this table mean?** That question has a name. It is called *grain*, and if you take only one word out of this whole course, take that one. By Friday you will have modeled a retail sales domain from business questions backward to a star schema, loaded it, and implemented a Type-2 slowly-changing dimension whose entire history you can reconstruct with one `SELECT`. Everything runs in Docker against **Postgres 16** — open source, no cloud account, no vendor lock-in (<https://hub.docker.com/_/postgres>).

The first thing to internalize is that **a data engineer owns a different failure surface than the people around them**. The backend engineer owns the transactional database that records that an order happened; when an `INSERT` fails or a row is locked, they are paged. The analyst owns the dashboard and the `SELECT` behind it; when the number on the dashboard is *wrong*, they are paged. The data scientist owns the model; when the features arrive late or malformed, their training run silently degrades. The data engineer owns the *pipeline between all of them* — the extract, the load, the model, the transformation, the table contract — and is paged when the dashboard is *stale*, when a load *double-counted*, when a column *changed shape upstream and nobody told the warehouse*. Section 1 of Lecture 1 draws the on-call boundary explicitly, because half of being a data engineer is knowing which pager is yours. The canonical mental model for this division of labor is in Kleppmann's *Designing Data-Intensive Applications* (<https://dataintensive.net/>), which separates systems of record from derived data systems — a distinction we will lean on all week.

The second thing to internalize is the **OLTP / OLAP split**. Online Transaction Processing (OLTP) is the world of the backend engineer: many small, concurrent reads and writes, each touching a handful of rows, normalized to avoid update anomalies, indexed for point lookups. Online Analytical Processing (OLAP) is the world we are building: few enormous reads, each scanning millions of rows to compute an aggregate, deliberately *denormalized* so a question can be answered without a ten-table join. The same business — a retailer — needs both: an OLTP database to record that order #88213 was placed at 14:32, and an OLAP warehouse to answer "what were weekly revenue trends by product category across all stores last quarter." Postgres can play both roles on a laptop, which is exactly why it is our teaching engine. The PostgreSQL 16 documentation is your reference all week (<https://www.postgresql.org/docs/16/index.html>); Lecture 1 shows the *same* retail data modeled once as a normalized OLTP schema and once as a denormalized OLAP star, side by side, so you can feel the difference in your hands.

The third thing to internalize is the **warehouse → lake → lakehouse lineage**, because you will be asked about it in every data-engineering interview you ever take. The *data warehouse* (Inmon and Kimball, 1990s) is a structured, schema-on-write store optimized for analytical SQL — the star schema you build this week is the warehouse pattern in miniature. The *data lake* (Hadoop era, 2010s) was the reaction: dump everything as raw files into cheap object storage, schema-on-read, defer structure until query time — flexible, but it rotted into "data swamps" with no transactions, no schema enforcement, and no reliable way to know what was current. The *lakehouse* (the present) is the synthesis: keep the cheap object storage and open file formats of the lake, but add the ACID transactions, schema enforcement, and table semantics of the warehouse on top, via table formats like Apache Iceberg and Delta Lake. We do not build a lakehouse this week — that is Phase II — but you cannot reason about where you are going without the lineage, so Lecture 1 traces it with dates, motivations, and the failure that birthed each generation. Kleppmann's storage-and-retrieval chapters (<https://dataintensive.net/>) underpin the columnar-vs-row argument that makes the warehouse fast in the first place.

The fourth thing to internalize is **Kimball's four-step dimensional design process**, which is the spine of Lecture 2 and the single most reusable skill in the week. Ralph Kimball's *The Data Warehouse Toolkit* (3rd edition, Wiley, with Margy Ross) lays it out as a discipline you run every single time you model a business process: (1) **select the business process** you are measuring — "retail sales", not "the sales department"; (2) **declare the grain** — what exactly one fact row represents, stated in one sentence with no "and"; (3) **identify the dimensions** — the descriptive "who / what / where / when / how" context that hangs off that grain; (4) **identify the facts** — the numeric measurements you will aggregate. Run those four steps in order and the schema falls out almost mechanically. Skip step 2 and you will build a table nobody can trust. The Kimball Group's freely published "Dimensional Modeling Techniques" reference is the canonical online companion to the book (<https://www.kimballgroup.com/data-warehouse-business-intelligence-resources/kimball-techniques/dimensional-modeling-techniques/>).

The fifth thing to internalize is **why grain is THE word**. The grain is the contract for a fact table: it fixes the level of detail, it determines which dimensions can attach, it determines whether a measure is safe to `SUM`, and it determines whether two different people querying the table get the same number. A fact table at the *line-item* grain (one row per product per order) can answer questions about individual products; a fact table at the *order* grain (one row per order) cannot, but it can never accidentally double-count an order total either. There is no universally "right" grain — there is only the *finest grain your source data supports*, which Kimball recommends you choose by default because you can always aggregate up from fine grain and never down from coarse. Lecture 2 spends real pages on this, with the additive / semi-additive / non-additive fact taxonomy that grain forces on you (a count is additive across every dimension; an account balance is additive across everything *except* time; a ratio is additive across nothing). PostgreSQL's `CREATE TABLE` reference is the tool you express grain with (<https://www.postgresql.org/docs/16/sql-createtable.html>).

The sixth thing to internalize is **star versus snowflake**. A *star schema* keeps each dimension as a single, wide, denormalized table — `dim_product` carries the category name and the brand name as plain text columns, repeated on every product row. A *snowflake schema* normalizes those out — `dim_product` points to `dim_category` points to `dim_department`, third-normal-form style. The star wins for analytical query simplicity and join performance (one hop from fact to any dimension); the snowflake saves storage and enforces hierarchy integrity at the cost of more joins. Kimball is famously opinionated here: prefer the star, snowflake only when a dimension is genuinely enormous or a hierarchy must be enforced relationally. Challenge 1 makes you build *both* and measure the query-cost difference yourself with `EXPLAIN ANALYZE` (<https://www.postgresql.org/docs/16/sql-explain.html>) so the trade-off is something you have felt, not just read.

The seventh thing to internalize is the **surrogate key**. A *natural key* is the business identifier — a product SKU, a customer email, a store code. A *surrogate key* is a meaningless integer the warehouse mints itself, generated with `GENERATED ALWAYS AS IDENTITY` (<https://www.postgresql.org/docs/16/ddl-identity-columns.html>). Every dimension in a Kimball star uses a surrogate key as its primary key and carries the natural key as an ordinary attribute. This is not bureaucracy — it is the mechanism that makes the *next* concept possible. A surrogate key decouples the warehouse row from the source identifier, survives the source re-using or recycling an identifier, joins faster as a narrow integer, and — crucially — lets a single natural entity have *multiple* dimension rows over time. Which brings us to the climax of the week.

The eighth thing to internalize is the **slowly-changing dimension (SCD)**, and specifically **Type 2**, which is the technical centerpiece of Lab 01. A dimension attribute changes over time: a customer moves to a new city, a product is re-categorized, a store changes region. The question is what the warehouse does with that change, and Kimball gives it numbered answers. **Type 0**: never change it (the attribute is immutable, like a date of birth). **Type 1**: overwrite the old value (you do not care about history; the dashboard should always show the current city). **Type 2**: *keep both rows* — close the old row with a `valid_to` date and `is_current = false`, open a new row with a fresh surrogate key, a `valid_from` date, and `is_current = true`. **Type 3**: keep a "previous value" column alongside the current one (you care about exactly one prior state). Type 2 is the one that earns the "data engineer" title, because it is the one that lets you answer "what category was this product in *at the time of the sale*" — a question a Type-1 overwrite can never answer again. Lecture 3 and Lab 01 implement Type 2 end to end: effective-dated rows, the `MERGE` (<https://www.postgresql.org/docs/16/sql-merge.html>) that closes the old version and opens the new in one statement, and a point-in-time audit query that reconstructs the dimension as it stood on any past date.

The ninth thing to internalize is the **conformed dimension**, the idea that makes a warehouse a *warehouse* and not a pile of disconnected tables. A conformed dimension is one shared, identical dimension (`dim_date`, `dim_product`, `dim_store`) used by *multiple* fact tables at *different* grains. Because `fact_sales` (transaction grain) and `fact_inventory_snapshot` (daily-snapshot grain) both join to the *same* `dim_product` and the *same* `dim_store`, you can write one query that drills across both — "for each product, what was revenue versus units on hand" — even though the two facts were loaded by completely separate pipelines. Conformance is what lets an organization integrate, rather than re-implementing "store" four times with four incompatible definitions. Challenge 2 makes you build a second fact table that conforms to Lab 01's dimensions and drill across both. The Kimball Group's enterprise-data-warehouse-bus-matrix technique is the formal statement of this idea (<https://www.kimballgroup.com/data-warehouse-business-intelligence-resources/kimball-techniques/dimensional-modeling-techniques/>).

By the end of the week you will be the person who, handed a vague business question and a pile of source tables, can stand at a whiteboard and say "the business process is retail sales, the grain is one row per product per order line, the dimensions are date / product / store / customer, the facts are quantity and extended amount, and we will track product re-categorization as a Type-2 SCD" — and then turn that sentence into running, queryable, auditable Postgres before lunch.

## Learning objectives

By the end of this week, you will be able to:

- **Distinguish** what a data engineer owns from what an analyst, data scientist, and backend engineer own, and name who is paged when each layer of the platform breaks. Ground the systems-of-record-vs-derived-data distinction in Kleppmann's *Designing Data-Intensive Applications* (<https://dataintensive.net/>).
- **Explain** OLTP versus OLAP from the access pattern up — many small writes versus few enormous reads, normalized versus denormalized, row-store versus column-store — and demonstrate the difference by modeling the same retail data both ways in Postgres (<https://www.postgresql.org/docs/16/index.html>).
- **Trace** the warehouse → lake → lakehouse lineage with the motivation and the failure mode that produced each generation, and place the star schema you build this week correctly within it.
- **Run** Kimball's four-step dimensional design process — select process, declare grain, identify dimensions, identify facts — on a new business process, citing the Kimball Group "Dimensional Modeling Techniques" reference (<https://www.kimballgroup.com/data-warehouse-business-intelligence-resources/kimball-techniques/dimensional-modeling-techniques/>).
- **Choose** the correct grain for a fact table, state it in one sentence with no "and", and defend it; classify each measure as additive, semi-additive, or non-additive.
- **Build** a retail star schema in Postgres 16 with `CREATE TABLE` (<https://www.postgresql.org/docs/16/sql-createtable.html>): `dim_date`, `dim_product`, `dim_store`, `dim_customer`, and `fact_sales`, with surrogate-key primary keys generated by `GENERATED ALWAYS AS IDENTITY` (<https://www.postgresql.org/docs/16/ddl-identity-columns.html>).
- **Implement** a Type-2 slowly-changing dimension with effective-dated rows (`valid_from`, `valid_to`, `is_current`) and the `MERGE` that closes the old version and opens the new (<https://www.postgresql.org/docs/16/sql-merge.html>), and audit it with a single point-in-time query.
- **Compare** star and snowflake schemas by normalizing one dimension and measuring the query-cost difference with `EXPLAIN ANALYZE` (<https://www.postgresql.org/docs/16/sql-explain.html>), then defend the trade.
- **Design** conformed dimensions shared across two fact tables at different grains and write a query that drills across both via the conformed keys.
- **Cite** Kimball & Ross's *The Data Warehouse Toolkit* (3rd ed.), the PostgreSQL 16 documentation, and Kleppmann's *Designing Data-Intensive Applications* for each technique covered.

## Topics covered

- **The data-engineering role boundary.** Data engineer vs analyst vs data scientist vs backend engineer; systems of record vs derived data systems; who is paged when each layer breaks.
- **OLTP vs OLAP.** Access patterns, normalization vs denormalization, row vs columnar storage, why one engine cannot be optimal for both at scale, and why Postgres is fine for both at laptop scale.
- **The storage lineage.** Data warehouse (schema-on-write) → data lake (schema-on-read) → lakehouse (open formats + ACID table semantics); the failure that produced each.
- **Kimball's four-step design process.** Select the business process; declare the grain; identify the dimensions; identify the facts — in that order, every time.
- **Fact-table grain.** Why grain is the contract; choosing the finest available grain; the consequences of getting it wrong; additive / semi-additive / non-additive facts.
- **Dimensions.** Descriptive attributes, the wide denormalized dimension, the role-playing dimension, the degenerate dimension (a key with no dimension table, like an order number), the factless fact table.
- **Star vs snowflake.** Denormalized single-hop dimensions vs normalized hierarchies; query simplicity and join cost vs storage and integrity; when each is correct.
- **Surrogate keys vs natural keys.** Why the warehouse mints its own meaningless integer key; `GENERATED ALWAYS AS IDENTITY`; how surrogate keys make Type-2 SCDs possible.
- **Slowly-changing dimensions.** Type 0 (retain original), Type 1 (overwrite), Type 2 (add a new effective-dated row), Type 3 (add a prior-value column); when each is appropriate.
- **Type-2 SCD mechanics.** `valid_from` / `valid_to` / `is_current`, the close-the-old-open-the-new `MERGE`, the single-query point-in-time audit.
- **Conformed dimensions.** One dimension shared by many facts at different grains; drilling across facts; the enterprise bus matrix.
- **The worked example: a retail sales warehouse.** Modeled from business questions backward to a star schema, loaded in Postgres, with a Type-2 product dimension, all in Docker.

## The "state the grain" promise

C27 has one contract that starts in Week 1 and never lets up: **every fact table you ship has its grain written down, in one sentence, with no "and", in a comment at the top of its `CREATE TABLE`.** A fact table without a declared grain is a fact table nobody can trust, because two engineers will read two different meanings into the same row and produce two different numbers from the same query. The rule, restated every week the rest of the course: *the grain is the contract, and an undocumented contract is no contract at all.*

We add a second promise this week, the **audit promise**: every Type-2 dimension you build can answer "what did this entity look like on date X" with a single `SELECT`, no application code, no spreadsheet, no archaeology. If your SCD cannot be audited from SQL alone, it is not done.

We use only standard SQL features available in PostgreSQL 16: `GENERATED ALWAYS AS IDENTITY` for surrogate keys, `MERGE` for the SCD upsert, generated columns for derived attributes, and `EXPLAIN ANALYZE` for cost.
