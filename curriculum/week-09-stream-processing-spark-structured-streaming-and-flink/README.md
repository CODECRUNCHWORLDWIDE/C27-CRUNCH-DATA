# Week 9 — Stream Processing (Spark Structured Streaming & Flink)

In Week 8 you stood up Apache Kafka in Docker and produced a synthetic clickstream into a topic called `clickstream` — keyed by `user_id`, partitioned across three partitions, every record an Avro-serialized event (`user_id`, `session_id`, `page`, `event_type`, `event_ts`, `processing_ts`) carrying a Confluent-style schema-registry magic byte plus schema id. You proved the producer was idempotent, watched the consumer-group offsets advance, and reasoned about partition assignment and rebalancing. That topic is now a **bounded-looking but logically unbounded** log: it keeps growing, the events keep arriving slightly out of order because the producer injects a configurable lateness, and nobody is reading it for analytics yet. This week you build the reader. By Friday you will have a Spark Structured Streaming job that consumes that exact `clickstream` topic, deserializes the Avro payload, stamps each event with its **event time**, applies a **watermark**, computes a **windowed count of events per page per 5-minute tumbling window**, correctly folds in a deliberately injected late event, and writes the result **exactly once** into an Iceberg (or Delta) table that the Week-6 dashboard can `SELECT` from. Then you build the same windowed count in **Apache Flink** with PyFlink and measure how the two systems differ in latency, semantics, and operational weight.

The framing for the whole week is one sentence: **streaming is batch's hard generalization.** Everything you learned about batch in Week 3 — incremental ingestion gated by a high-water mark, late-arriving records that show up after you thought a partition was closed, idempotent upserts so a re-run does not double-count — has an exact streaming analog, and the streaming analog is *the same idea made continuous and unbounded*. In Week 3 you ran an Airflow DAG once an hour, read everything newer than a stored `last_loaded_ts` watermark, and `MERGE`d it into a Postgres table on a natural key so a retry was a no-op. Replace "once an hour" with "every micro-batch," replace "stored watermark column" with "the streaming engine's watermark," replace "late record I have to backfill" with "late event the watermark has not yet expired," and replace "`MERGE` on natural key" with "`foreachBatch` that does the same `MERGE`," and you have Structured Streaming. The mechanics are isomorphic. The hard part is that in batch the boundaries of the dataset are *given to you* (the file exists, the day is over, you know when you are done); in streaming the boundaries are *something you must infer from the data itself*, because the stream never ends and you can never be sure no later event is still in flight. The watermark is precisely that inference.

So the first concept to pin down is **event time versus processing time**. Event time is *when the thing happened* — the `event_ts` your Week-8 producer stamped at the moment the user clicked. Processing time is *when your engine saw the record* — wall-clock time at the executor when the micro-batch read it off Kafka. These two clocks drift apart constantly: a mobile client buffers events offline and flushes them ten minutes later; a Kafka partition lags under load; a consumer restarts and replays. If you window by processing time you get answers that are easy to compute and *wrong for analytics*, because the same user click lands in different windows depending on how congested your pipeline was. If you window by event time you get answers that are *correct and reproducible* — the click at 09:03:17 always belongs to the 09:00–09:05 window no matter when it arrives — but you pay for that correctness with a hard question: **how long do you wait for stragglers before you declare a window final and emit its result?** That is the watermark.

A **watermark** is the streaming engine's running assertion: "I do not expect to see any event with an event time earlier than *T* anymore." Concretely, in Spark you write `withWatermark("event_ts", "10 minutes")`, and Spark tracks the maximum event time it has observed so far and subtracts ten minutes; that difference is the current watermark. Any window whose end is older than the watermark is considered *closed* — its state can be dropped, its result emitted, and any event arriving for it afterward is *too late* and discarded. This is exactly the Week-3 high-water mark, except Week 3's watermark was a single value you stored in a control table and advanced by hand at the end of each run, and the streaming watermark is a value the engine computes and advances continuously from the data as it flows. The trade-off is the same one Week 3 made explicit: a longer watermark delay tolerates more lateness (fewer dropped events) at the cost of higher latency and more state held in memory; a shorter delay emits results faster and uses less state but drops more stragglers. There is no free lunch — the watermark is a knob on the completeness/latency curve, and the right setting is a business decision, not a technical one.

With event time and watermarks in hand, the rest is **windowing and aggregation**. A **tumbling window** chops event time into fixed, non-overlapping intervals — 5-minute buckets, each event in exactly one — and is what the lab uses. A **sliding window** overlaps — a 10-minute window that advances every 5 minutes, so each event lands in two windows — and is what you reach for when you want a smoothed moving metric. A **session window** has no fixed size at all: it groups events by *gaps* of inactivity, closing a user's session when they have been quiet for, say, fifteen minutes, which is exactly how you would reconstruct browsing sessions from the clickstream. All three are stateful: the engine must hold partial aggregates (running counts, running sums) in a **state store** between micro-batches, keyed by the grouping key and the window. The watermark is what bounds that state — without it, the engine would have to remember every window forever, because it could never prove no late event is coming. This is why **append output mode requires a watermark**: append only emits a window's final result once, and "final" is only definable once the watermark has passed the window's end.

That brings up **output modes**, the part everyone gets wrong first. Structured Streaming has three. **Complete mode** re-emits the *entire* result table every micro-batch — every window, every key, every time — which is correct but only feasible for small aggregates and requires no watermark because nothing is ever finalized or dropped. **Update mode** emits only the rows that *changed* since the last micro-batch — a window's count goes from 3 to 4, you emit the new 4 — which is efficient and is what you want for a live dashboard that upserts. **Append mode** emits each result row *exactly once and never again*, only after the watermark guarantees it will not change, which is what you want when writing immutable rows to a downstream sink that cannot handle updates. The rule to memorize: append on an aggregation is **only legal with a watermark**, because append needs a definition of "this row is final," and the watermark is that definition. Complete mode does not need one; update mode tolerates one. We map each mode to a Week-3 batch analog: complete is a full table rebuild, update is an upsert, append is an insert-only incremental load.

The payoff concept is **exactly-once delivery into a lakehouse table**, and it is *not* magic — it is two ingredients multiplied together. Ingredient one is a **replayable source**: Kafka is replayable because every record has a durable offset, so if a micro-batch fails after reading but before writing, the engine can re-read the exact same records by rewinding to the committed offset. Structured Streaming records the offset range of each micro-batch in its **checkpoint**, a directory (we put it on MinIO) holding the write-ahead log of which offsets each batch covered plus the contents of the state store. Ingredient two is an **idempotent or transactional sink**: the write must be safe to repeat, so that replaying a batch after a crash does not double-count. A Delta or Iceberg `MERGE` keyed on `(window, page)` is idempotent — replaying it overwrites the same rows rather than appending duplicates — which is *exactly* the Week-3 idempotent upsert, now invoked from inside `foreachBatch`. Exactly-once is the product: **replayable source × checkpoint × idempotent sink**. Drop any one factor and you fall back to at-least-once (duplicates possible) or at-most-once (loss possible). This is the single most important sentence in the week, and it is the same sentence Week 3 used about batch retries.

Writing the windowed aggregate into Iceberg or Delta is the **streaming-lakehouse pattern**: the stream lands continuously into a table that is *also* a normal analytics table queryable by DuckDB, Spark SQL, or the dashboard, with ACID guarantees, time travel, and schema evolution. The stream is the writer; the lakehouse table is the shared truth; everything downstream is a reader. This is the architecture most modern data platforms converge on — one table, continuously updated by a stream, read by both dashboards and batch jobs — and it dissolves the old "speed layer vs batch layer" Lambda split into a single continuously-updated table. We also confront the heresy that closes the lecture: **a 15-minute micro-batch often beats a true stream.** If your dashboard refreshes every fifteen minutes anyway, a Structured Streaming job triggered every fifteen minutes (or even Airflow running a batch job every fifteen minutes) is simpler to operate, cheaper to run, and easier to reason about than a sub-second streaming system whose latency nobody is consuming. Latency you do not need is cost you do pay. Choose true streaming when sub-second-to-seconds latency is a product requirement, not a vanity metric.

Finally, **Apache Flink as the dedicated alternative.** Spark Structured Streaming is a micro-batch engine wearing a streaming API: under the hood it still runs a sequence of small batch jobs, which caps its latency at roughly the micro-batch interval (hundreds of milliseconds to seconds) but inherits Spark's mature batch ecosystem and operational familiarity. Flink is a *true* record-at-a-time streaming engine: events flow through a standing dataflow graph of operators, processed one at a time as they arrive, giving single-digit-millisecond latency and a richer event-time and state model — at the cost of a separate runtime, a separate mental model, and more operational surface. Flink's exactly-once is built on **asynchronous barrier snapshotting** (a variant of the Chandy–Lamport distributed-snapshot algorithm) plus **two-phase-commit sinks**, where Spark's is built on checkpointed offsets plus idempotent writes — different mechanisms, same guarantee. You will build the identical windowed count in PyFlink, compare the wall-clock latency and the exactly-once mechanics, and develop a defensible answer to "Spark or Flink?" that is about the workload, not the hype. Week 10 then takes the table this week produces and asks whether the data in it is *correct* — data quality, the natural next question once the pipeline runs.

## Topics covered

- **Stream processing vs batch processing.** Unbounded vs bounded datasets; continuous vs one-shot computation; why the stream's boundaries must be inferred from the data while the batch's boundaries are given. The Dataflow Model's framing that batch is a special case of streaming where the window is "all of time."
- **Event time vs processing time vs ingestion time.** The three clocks, how and why they drift, the skew between them, and why analytics correctness demands event-time semantics. The classic "where in event time / when in processing time" two-axis picture from Streaming 102.
- **Watermarks and late data.** The watermark as a running lower-bound assertion on event time; `withWatermark`; allowed lateness; what "too late" means and what happens to a dropped event; the explicit tie-back to the Week-3 high-water mark.
- **Tumbling, sliding, and session windows.** Fixed non-overlapping vs overlapping vs gap-defined windows; `window(col, "5 minutes")`, `window(col, "10 minutes", "5 minutes")`, `session_window(col, "15 minutes")`; which metric calls for which window.
- **Stateful aggregation and the state store.** Partial aggregates held between micro-batches keyed by group and window; how the watermark bounds state size; state-store checkpointing; the RocksDB state-store backend for large state.
- **Output modes.** Append, update, complete; the legality matrix against watermarks and aggregations; mapping each mode to a Week-3 batch analog (insert-only, upsert, full rebuild).
- **Checkpointing.** The checkpoint directory contents (offset write-ahead log, commit log, state-store snapshots, metadata); why a checkpoint is tied to a specific query and is not portable across query-plan changes; placing the checkpoint on MinIO.
- **Exactly-once delivery.** The `replayable source × checkpoint × idempotent sink` multiplication; Kafka offsets as the replayable source; `foreachBatch` + Delta/Iceberg `MERGE` as the idempotent sink; at-least-once vs at-most-once as the failure modes when a factor is missing.
- **The streaming-lakehouse pattern.** Streaming continuously into an Iceberg/Delta table that is simultaneously a queryable ACID analytics table; the dissolution of the Lambda speed-layer/batch-layer split into one continuously-updated table; Delta and Iceberg streaming-sink semantics.
- **Trigger modes.** Default (process as fast as possible), `Trigger.ProcessingTime("15 minutes")` (fixed micro-batch), `Trigger.AvailableNow()` (drain all available data then stop, the streaming-as-batch trick); when a 15-minute micro-batch beats a true stream.
- **Apache Flink as the dedicated alternative.** JobManager/TaskManager architecture; true record-at-a-time dataflow vs Spark micro-batch; Flink event-time and watermarks; PyFlink Table API `WATERMARK FOR` + `TUMBLE`; Flink checkpointing via asynchronous barrier snapshotting (Chandy–Lamport) and two-phase-commit sinks; the latency/throughput/complexity trade vs Spark; when to pick which.

## A note on tone

C27 is written in **engineer's-logbook voice**. We pin versions ("Spark 3.5.1," "Delta 3.1.0," "Iceberg 1.5.0," "Flink 1.18.1," "PyFlink 1.18.1," "Kafka 3.7"). We cite behavior by the doc section that defines it, not by folklore. We say "the job processed 184,302 events across 37 micro-batches at a mean `processedRowsPerSecond` of 5,120, the watermark trailed the max event time by 10 minutes, the 09:00–09:05 window emitted a count of 412 for `/checkout`, and the injected late event at 09:02:41 (arriving at processing time 09:11:30, watermark 09:09:00) folded into that window because it landed inside the 10-minute allowance" — not "the streaming job worked." A streaming result is a count per window per key. If your write-up says "low latency" or "exactly-once" without a measured number and the mechanism that produced it, you have not written one yet.

## A note on data size and runtime

Streaming jobs are long-lived, not one-shot — that is the mental adjustment. Expect:

- The Week-8 `clickstream` topic at ~1,000 events/second for the lab is **~60 MB/minute** of Avro on the broker; you only ever read offsets forward, so disk on your laptop is fine for an hour-long run.
- A Structured Streaming **checkpoint** directory on MinIO grows to **a few MB** for this aggregate — it holds the offset write-ahead log, the commit log, and state-store snapshots, not the data.
- The **state store** for a 5-minute tumbling window over ~10 pages with a 10-minute watermark holds at most a few hundred keys — kilobytes. State is bounded by the watermark; that is the whole point.
- The output **Iceberg/Delta table** is one row per `(window, page)` — for an hour of data that is ~12 windows × ~10 pages = ~120 rows, well under a megabyte. The table is tiny; the stream feeding it is not.

Commit your job code, the `docker-compose.yml`, the checkpoint *location config* (not the checkpoint contents), and the PERF.md. Gitignore the checkpoint directory, the MinIO data volume, and any local Kafka logs.

## A note on running streaming jobs locally

A streaming query **does not return** — `query.awaitTermination()` blocks forever by design. To work interactively, run with a `Trigger.AvailableNow()` to drain-and-stop, or run the query in one terminal and inspect `query.lastProgress` / the Spark UI (`http://localhost:4040`) from another. To restart cleanly, stop the query, and **do not delete the checkpoint** unless you intend to lose exactly-once state — the checkpoint is what lets a restart resume from the committed offset. Deleting it and re-running re-reads from `startingOffsets` and can double-count. This footgun is the single most common Week-9 mistake; the exercises call it out explicitly.
