# Week 2 — Advanced SQL & Analytical Queries: Window Functions and Frames, CTEs and Recursive CTEs, GROUPING SETS / ROLLUP / CUBE, Anti-Joins and Semi-Joins, and Reading a Query Plan Before You Guess

Welcome to **C27 · Crunch Data**, Week 2. Last week you built a star schema: `dim_date`, `dim_product`, `dim_store`, `dim_customer`, and a `fact_sales` table at the grain of one row per product per sale, with a Type-2 slowly-changing dimension you can audit with a single query. The model is the contract; this week you learn to *interrogate* it. The grain you chose in Week 1 is what makes the queries in this week possible — every "rank within category," every "running 7-day total," every "month-over-month change" is a question that only has an answer because each fact row sits at a known, defensible grain. This week is the week the SQL stops being `SELECT * FROM table WHERE x` and starts being the analytical language that every real pipeline is written in.

The single most important idea this week is that **a window function does not collapse rows the way a `GROUP BY` does**. A `GROUP BY` takes many rows and returns one row per group; you lose the detail. A window function computes an aggregate *over a window of rows* and attaches the answer back to every individual row, so you keep the detail and get the aggregate next to it. "Show me each sale, and next to it the running total of sales for that store so far this month" is impossible with `GROUP BY` and trivial with a window function. The canonical reference is the PostgreSQL window-functions tutorial at <https://www.postgresql.org/docs/16/tutorial-window.html>, and you should have it open while you read Lecture 1. The mental model — *partition the rows, order them, define a frame, compute over the frame, attach the result to every row* — is the spine of the entire week.

The second idea is that the `OVER` clause has three independent knobs and you must hold them apart in your head. `PARTITION BY` divides the rows into groups (resets the computation at each group boundary, like `GROUP BY` but without collapsing). `ORDER BY` *inside the window* defines the order the rows are processed in, which is what makes "running total" and "previous row" meaningful. And the **frame clause** (`ROWS`/`RANGE`/`GROUPS BETWEEN ... AND ...`) defines, for the current row, exactly which neighbouring rows the aggregate sees. Most people learn `PARTITION BY` and `ORDER BY` and never learn the frame clause, and then they are mystified when a running `SUM` over an `ORDER BY` with ties gives a different answer than they expected — because the default frame is `RANGE UNBOUNDED PRECEDING AND CURRENT ROW`, which lumps tied rows together, and they wanted `ROWS`. The frame clause is the part of window functions that separates people who use them from people who *understand* them, and the functions reference at <https://www.postgresql.org/docs/16/functions-window.html> documents every built-in window function precisely.

The third idea is the family of ranking functions, and the fact that `ROW_NUMBER`, `RANK`, and `DENSE_RANK` are *three different answers to "what number is this row."* `ROW_NUMBER` gives every row a distinct integer with no ties ever (1, 2, 3, 4). `RANK` gives tied rows the same rank and then *skips* (1, 1, 3, 4). `DENSE_RANK` gives tied rows the same rank and does *not* skip (1, 1, 2, 3). The difference matters enormously in practice: "the top 3 products by revenue per category" is `DENSE_RANK <= 3` if you want up to three *distinct revenue levels* and `ROW_NUMBER <= 3` if you want exactly three rows. A quiz answer that confuses these is the single most common SQL-interview miss, and Lecture 1 drills it with the retail data.

The fourth idea is `LAG` and `LEAD`, the two functions that let a row see its neighbours. `LAG(metric, 1)` reads the value of `metric` from the row one position *before* the current row in the window's order; `LEAD` reads the row *after*. This is how every period-over-period calculation is written — "this month's sales minus last month's sales" is `current - LAG(current, 1) OVER (ORDER BY month)`. Before window functions existed, this required a self-join of the table to a shifted copy of itself, which was slow, error-prone at the boundaries, and unreadable. `LAG`/`LEAD` make it one line. You will use them in Exercise 1 and in at least three of the ten gauntlet questions.

The fifth idea is the **Common Table Expression**, the `WITH` clause, which is the tool that turns an unreadable 200-line nested subquery into a readable, top-to-bottom sequence of named steps. A CTE is a named subquery you define once at the top and reference by name below, and the discipline of "name every intermediate result" is the difference between SQL a colleague can review and SQL that gets rewritten from scratch because nobody can follow it. PostgreSQL 16's `WITH` queries are documented at <https://www.postgresql.org/docs/16/queries-with.html>, and a subtle point you must learn this week is that PostgreSQL lets you *control* whether a CTE is materialized (computed once into a temp result) or inlined into the main query, with the `MATERIALIZED` and `NOT MATERIALIZED` keywords — a knob that can make a query an order of magnitude faster or slower, covered in Lecture 2.

The sixth idea is the **recursive CTE**, which is the only way standard SQL can walk a hierarchy or generate a series. A recursive CTE has an *anchor* term (the starting rows), the keyword `UNION ALL`, and a *recursive* term that references the CTE's own name and runs again and again until it produces no new rows. The two canonical uses are walking a tree — a product-category hierarchy, an org chart, a bill-of-materials — and generating a series, like a complete date spine you can left-join your facts onto so that days with zero sales still appear. Lecture 2 walks both, against the retail schema. The reference is the recursive-query section of the same `WITH` page above.

The seventh idea is the **grouping-set family**: `GROUPING SETS`, `ROLLUP`, and `CUBE`. A normal `GROUP BY a, b` gives you one grouping. `GROUPING SETS ((a,b), (a), ())` gives you *three* groupings in one pass — the detail, the subtotal by `a`, and the grand total — all in a single result set. `ROLLUP(a, b)` is shorthand for the hierarchical subtotals `(a,b), (a), ()`; `CUBE(a, b)` is shorthand for every combination `(a,b), (a), (b), ()`. The catch is that the subtotal rows have `NULL` in the columns that were rolled up, and a real `NULL` in the data is indistinguishable from a subtotal `NULL` *unless* you use the `GROUPING()` function, which returns 1 for "this column is a subtotal here" and 0 otherwise. PostgreSQL documents the whole family at <https://www.postgresql.org/docs/16/queries-table-expressions.html#QUERIES-GROUPING-SETS>, and Lecture 2 shows you exactly how a finance report's "subtotal per region, plus grand total" is one `ROLLUP` query.

The eighth idea is the distinction between an **anti-join** and a **semi-join**, and the trap that lives between them. A semi-join answers "rows in A that have a match in B" — `WHERE EXISTS (SELECT 1 FROM B WHERE ...)` or `WHERE a.id IN (SELECT id FROM B)`. An anti-join answers "rows in A that have *no* match in B" — `WHERE NOT EXISTS (...)` or a `LEFT JOIN ... WHERE b.id IS NULL`. "Find customers who have never ordered" is the textbook anti-join. The trap is `NOT IN`: `WHERE id NOT IN (SELECT customer_id FROM fact_sales)` looks identical to `NOT EXISTS`, but if the subquery returns even one `NULL`, `NOT IN` returns *zero rows for everything*, because `x NOT IN (1, 2, NULL)` evaluates to `NULL` (unknown), never `TRUE`. This is one of the most expensive silent bugs in analytical SQL, and Lecture 2 makes you feel it on purpose.

The ninth idea, and the one that turns a SQL author into a SQL engineer, is **reading the query plan before you guess why a query is slow**. `EXPLAIN` shows you the plan the optimizer chose; `EXPLAIN ANALYZE` *runs* the query and shows you estimated rows next to actual rows, so you can see where the optimizer's guess was wrong. The single highest-value skill in this week is looking at a plan, finding the `Seq Scan` that is reading ten million rows to return ten, recognizing it should have been an `Index Scan`, and proving it with a `CREATE INDEX` and a re-measured `EXPLAIN ANALYZE`. The PostgreSQL `EXPLAIN` reference is at <https://www.postgresql.org/docs/16/sql-explain.html> and the indispensable "Using EXPLAIN" walkthrough is at <https://www.postgresql.org/docs/16/using-explain.html>. Lecture 3 reads real plans line by line, and Challenge 2 makes you tune one and report the before/after.

The tenth idea is that **the join is where queries explode**, and the optimizer's job is to estimate how many rows each join will produce so it can pick `Nested Loop` vs `Hash Join` vs `Merge Join` and order the joins correctly. When its row estimate is wrong — because statistics are stale, or because two columns are correlated in a way it can't model — it picks a `Nested Loop` that should have been a `Hash Join`, the inner side gets scanned millions of times, and a query that should take 50ms takes 50 seconds. Lecture 3 teaches you to spot "estimated 10 rows, actual 4,000,000 rows" in an `EXPLAIN ANALYZE` and recognize it as the smoking gun. The storage and index internals behind all of this — B-trees, why a sequential scan is sometimes *correct*, how a hash join builds and probes — are covered with reference to Martin Kleppmann's *Designing Data-Intensive Applications* at <https://dataintensive.net/>.

A word on `QUALIFY`. You will hear about a clause called `QUALIFY` that filters on a window-function result directly — `SELECT ... QUALIFY ROW_NUMBER() OVER (...) = 1`. It is genuinely lovely, and it exists in **DuckDB, Snowflake, and BigQuery**. It does **not** exist in PostgreSQL, and it will never compile there. This week teaches `QUALIFY` honestly: what it does, where it works, and the exact PostgreSQL-equivalent pattern you must use instead — wrap the window function in a subquery or CTE and filter on its alias in the outer query, because you cannot reference a window function in the `WHERE` of the same query level (the `WHERE` runs *before* window functions are computed). DuckDB's `QUALIFY` page is at <https://duckdb.org/docs/sql/query_syntax/qualify> and the DuckDB docs root is <https://duckdb.org/docs/>. You will run the same logic both ways and see them produce identical results.

By the end of this week you will be the person on your team who, handed a vague business question and a slow query, can turn the question into correct analytical SQL, read the plan to find why it is slow, and make it measurably faster — and who knows which clauses are PostgreSQL and which are DuckDB before they ship the query to the wrong engine.

## Learning objectives

By the end of this week, you will be able to:

- **Write** window functions over the retail star schema with `PARTITION BY`, in-window `ORDER BY`, and an explicit frame clause, and explain why a window function keeps detail while `GROUP BY` collapses it. Cite the PostgreSQL window tutorial at <https://www.postgresql.org/docs/16/tutorial-window.html>.
- **Distinguish** `ROW_NUMBER`, `RANK`, and `DENSE_RANK` precisely, and pick the right one for "top-N per group" depending on whether ties should consume a slot. Cite the window-functions reference at <https://www.postgresql.org/docs/16/functions-window.html>.
- **Compute** running and cumulative aggregates and period-over-period change with `LAG`/`LEAD` and a correct frame (`ROWS BETWEEN ... AND ...`), and explain the difference between `ROWS`, `RANGE`, and `GROUPS` frames.
- **Author** Common Table Expressions, control materialization with `MATERIALIZED` / `NOT MATERIALIZED`, and write a recursive CTE for both a category hierarchy and a generated date spine. Cite <https://www.postgresql.org/docs/16/queries-with.html>.
- **Produce** subtotal-and-grand-total reports in one pass with `GROUPING SETS`, `ROLLUP`, and `CUBE`, and use the `GROUPING()` function to tell a subtotal `NULL` from a data `NULL`. Cite <https://www.postgresql.org/docs/16/queries-table-expressions.html#QUERIES-GROUPING-SETS>.
- **Write** anti-joins (`NOT EXISTS`, `LEFT JOIN ... IS NULL`) and semi-joins (`EXISTS`, `IN`), and explain precisely why `NOT IN` with a `NULL` in the subquery returns zero rows.
- **Read** an `EXPLAIN ANALYZE` plan: estimated vs actual rows, cost units, `Buffers`, and the node types (`Seq Scan`, `Index Scan`, `Bitmap Heap Scan`, `Nested Loop`, `Hash Join`, `Merge Join`, `Sort`, `Aggregate`). Cite <https://www.postgresql.org/docs/16/using-explain.html>.
- **Diagnose** a slow query — a sequential scan that should be an index seek, a join whose cardinality estimate exploded, a sort that spilled to disk — add the right index, and prove the improvement with a re-measured plan. Cite <https://www.postgresql.org/docs/16/indexes.html>.
- **Translate** `QUALIFY` (DuckDB / Snowflake / BigQuery) into the PostgreSQL subquery-on-window pattern and back, and explain why `WHERE` cannot reference a window function. Cite <https://duckdb.org/docs/sql/query_syntax/qualify>.
- **Compare** a PostgreSQL row-oriented plan to a DuckDB columnar / vectorized plan for the same analytical query, and explain why the columnar engine reads fewer bytes. Cite <https://duckdb.org/docs/guides/meta/explain> and Kleppmann at <https://dataintensive.net/>.
- **Cite** PostgreSQL 16 documentation, DuckDB documentation, and *Designing Data-Intensive Applications* for every technique you use this week.

## Topics covered

- **Window functions.** The `OVER` clause; `PARTITION BY`; in-window `ORDER BY`; the distinction from `GROUP BY` (keep detail vs collapse).
- **Ranking functions.** `ROW_NUMBER`, `RANK`, `DENSE_RANK`, `NTILE`, `PERCENT_RANK`, `CUME_DIST`; top-N-per-group; ties.
- **Offset functions.** `LAG`, `LEAD`, `FIRST_VALUE`, `LAST_VALUE`, `NTH_VALUE`; period-over-period change; the boundary-NULL question.
- **Frames.** `ROWS` vs `RANGE` vs `GROUPS`; `BETWEEN ... AND ...`; `UNBOUNDED PRECEDING`, `CURRENT ROW`, `UNBOUNDED FOLLOWING`; the default frame and the surprise it causes; running and centered moving aggregates.
- **CTEs.** `WITH`; naming intermediate steps; PostgreSQL `MATERIALIZED` / `NOT MATERIALIZED`; the optimization-fence change in Postgres 12+.
- **Recursive CTEs.** Anchor + `UNION ALL` + recursive term; a category-tree walk; a generated date spine; termination and cycle detection.
- **Grouping sets.** `GROUPING SETS`, `ROLLUP`, `CUBE`; the subtotal `NULL`; the `GROUPING()` function; one-pass subtotal reports.
- **Set-membership joins.** Semi-joins (`EXISTS`, `IN`); anti-joins (`NOT EXISTS`, `LEFT JOIN ... IS NULL`); the `NOT IN`-with-`NULL` trap; relational division ("sold in every store").
- **`QUALIFY`.** What it is, where it works (DuckDB / Snowflake / BigQuery), why PostgreSQL lacks it, and the exact equivalent.
- **Reading a plan.** `EXPLAIN`, `EXPLAIN ANALYZE`, `EXPLAIN (ANALYZE, BUFFERS)`; estimated vs actual rows; cost units; the node types.
- **Indexes.** B-tree indexes; composite (multi-column) indexes and column order; partial indexes; covering / `INCLUDE` indexes; `Index Only Scan`; when a `Seq Scan` is correct.
- **Join algorithms.** `Nested Loop`, `Hash Join`, `Merge Join`; their cost shapes; the join that explodes when cardinality estimation is wrong.
- **DuckDB for comparison.** Columnar / vectorized execution; `EXPLAIN ANALYZE` in DuckDB; why the same query reads fewer bytes; running PostgreSQL-equivalent logic with `QUALIFY`.

## The "read the plan before you guess" promise

C27 treats one sentence as a contract for the rest of the course: **you do not get to say a query is slow because of X until you have read the plan and seen X.** "I think it's the join" is a guess; `EXPLAIN ANALYZE` showing a `Nested Loop` with `actual rows=4012398` where the estimate said `rows=11` is evidence. Every tuning claim you make this week — in the gauntlet, in both challenges, in the mini-project — must be backed by a captured plan, before and after. A pull request that says "made the query faster" without a plan capture is, by definition, a pull request that has not been reviewed.

We add a portability contract too: **every query you ship is labelled with the engine it targets.** `QUALIFY` is DuckDB / Snowflake / BigQuery; `MATERIALIZED` on a CTE is PostgreSQL; `GROUPING SETS` is both. Knowing which clause belongs to which engine *before* you paste it into a production warehouse is the difference between a query that runs and a 2am syntax error.
