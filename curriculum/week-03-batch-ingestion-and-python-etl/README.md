# Week 3 — Batch Ingestion and Python ETL: Watermarked Incremental Loads, Idempotent Upserts, COPY Bulk Loading, Late-Arriving Records, and the Pipeline You Can Re-Run at 3 AM Without Thinking

Welcome to **C27 · Crunch Data**, Week 3. Week 1 gave you a star schema — `dim_customer`, `dim_product`, `dim_date`, and a `fact_sales` grain you can defend — and a Type-2 slowly-changing dimension you can audit with one query. Week 2 taught you to interrogate that warehouse with window functions, recursive CTEs, grouping sets, and `EXPLAIN ANALYZE`. Both weeks assumed the data was already *in* the warehouse. This week we answer the question that was hiding under that assumption the whole time: **how did the data get there, and how do you load tomorrow's data without corrupting today's?** The technology is **Python 3.12**, **psycopg 3** (the current generation of the Postgres driver, `pip install "psycopg[binary]"` — see <https://www.psycopg.org/psycopg3/docs/>), and **PostgreSQL 16** in Docker. The single question that carries the whole week is the one a tired on-call engineer asks at 3 AM: *if I run this job again, will it double-count?*

The first thing to internalize is the difference between **ETL and ELT**, because the rest of the week depends on it. ETL — extract, transform, load — pulls data out of a source, reshapes it in a separate compute layer (historically a dedicated ETL server, today a Python process or a Spark job), and writes the finished, conformed result into the warehouse. ELT — extract, load, transform — dumps the raw source into the warehouse first and does the reshaping *inside* the warehouse with SQL (the pattern dbt popularized, which you will meet in Week 5). ELT won the lakehouse era because cloud warehouses made compute cheap and elastic, so it is cheaper to land raw bytes and transform them with the same engine that queries them than to operate a separate transform tier. But ETL never died: when the transform must happen *before* the data is allowed to land (PII redaction, schema enforcement at the boundary, a join against a system the warehouse cannot reach), you transform in flight. This week you build an ETL job, because doing the transform in Python first forces you to confront every failure mode explicitly; Week 5's dbt rebuild will show you the ELT mirror image. The canonical framing of the trade-off is Kleppmann's *Designing Data-Intensive Applications*, chapter 10 on batch processing <https://dataintensive.net/>.

The second thing to internalize is the **anatomy of a batch job**: extract → stage → transform → load → publish. You extract from the source into a *staging table* — a landing zone, usually a plain unconstrained table, that holds the raw extracted rows for this run and nothing else. You transform the staged rows (type-cast, deduplicate, conform keys) either in Python or in SQL against the staging table. You load the transformed rows into the warehouse target with an *upsert* so a re-run overwrites rather than appends. You publish — make the new data visible to readers, advance the watermark, log the run. Each stage is a place a job can die, and the staging-table pattern is what lets it die safely: a half-finished extract corrupts only the staging table, which you truncate and refill on the next run, never the warehouse target. PostgreSQL's `TRUNCATE` and `INSERT` semantics are documented at <https://www.postgresql.org/docs/16/sql-truncate.html> and <https://www.postgresql.org/docs/16/sql-insert.html>.

The third thing to internalize is **full versus incremental loads**. A full load reads the entire source every run and replaces the entire target — simple, always correct, and ruinously expensive once the source has a hundred million rows. An incremental load reads only the rows that changed since the last successful run and merges just those into the target. Incrementality is the whole game in production data engineering, and it has exactly one hard prerequisite: a reliable way to ask the source "what changed?" That mechanism is the **high-water mark**, or **watermark** — a stored value, usually `max(updated_at)` or `max(id)` from the previous run, that the next run reads forward from. You keep a `--full-refresh` escape hatch for the day the incremental logic is suspect, but you run incrementally every other day. Kimball's writing on incremental fact loading is the dimensional-modeling reference here <https://www.kimballgroup.com/data-warehouse-business-intelligence-resources/kimball-techniques/dimensional-modeling-techniques/>.

The fourth thing to internalize is **idempotency**, the property the lecture title promises and the property the whole week is built to earn. An operation is idempotent if applying it once and applying it five times produce the same result — the way `x = 5` is idempotent but `x += 5` is not. A pipeline is idempotent if re-running it over the same input leaves the warehouse in exactly the same state as running it once. This is not a nicety; it is the difference between a pipeline you can retry and a pipeline that loses you a night's sleep. Orchestrators retry failed tasks automatically (you will wire one up in Week 4); networks drop mid-write; an engineer fat-fingers a manual re-run. If any of those double-counts your revenue, the pipeline is broken by design. The mechanism that buys idempotency is the **upsert**: `INSERT ... ON CONFLICT ... DO UPDATE`, documented at <https://www.postgresql.org/docs/16/sql-insert.html#SQL-ON-CONFLICT>, which inserts a row if its key is new and overwrites it if the key already exists, so re-loading the same row is a no-op rather than a duplicate.

The fifth thing to internalize is **restartability**, idempotency's sibling. A restartable job is one that, killed at any instant — `kill -9`, an OOM, a pulled network cable — leaves the system in a consistent state you can simply re-run from. The two enabling disciplines are *atomic batches* (each batch of work commits in a single transaction, so a crash mid-batch rolls the whole batch back rather than leaving half of it applied) and *transactionally-stored watermarks* (you advance the watermark **in the same transaction** that loads the data it covers, so a crash can never leave the watermark pointing past data that was never loaded). Get this wrong — advance the watermark in a separate statement after the load commits — and a crash in the gap silently skips a batch forever. PostgreSQL's transaction semantics are at <https://www.postgresql.org/docs/16/tutorial-transactions.html>.

The sixth thing to internalize is the **late-arriving record**, the failure mode that humbles every naive watermark. Records have at least two timestamps: *event time* (when the thing happened — the order was placed) and *ingestion/processing time* (when your pipeline saw it). Usually they are close. Sometimes they are not: a mobile app buffers events offline and uploads them three days later; a partner sends a corrected invoice for last Tuesday; a clock skew backdates a record. If your watermark is on `event_time` and you read `WHERE event_time > last_watermark`, a record stamped three days ago that arrives today is *behind* the watermark and gets silently dropped — the worst kind of bug, because nothing errors. The fixes are real and this week teaches all of them: watermark on an ingestion-time or monotonic-`id` column the source controls (so late events still sort after the watermark), use a **lookback window** that re-reads the last N days every run, and upsert on the record's natural key so a re-seen record *corrects* the aggregate instead of duplicating it. Kleppmann's chapter 11 on stream processing develops event-time-versus-processing-time in depth <https://dataintensive.net/>, and Week 9's streaming watermarks are the same idea generalized.

The seventh thing to internalize is **bulk loading with `COPY`**. There is a right way to move a million rows into Postgres and two wrong ways. The first wrong way is a million single-row `INSERT` statements — each is a network round trip and its own implicit transaction, and you will measure this at hundreds of rows per second. The second wrong way is one gigantic `INSERT` (or one `executemany` without batching) that buffers the entire dataset in memory and in one transaction's WAL before anything commits — it works until the dataset outgrows RAM, then it falls over. The right way is `COPY`, Postgres's bulk-load path, which streams rows directly into the table with minimal per-row overhead and is typically an order of magnitude faster than `INSERT`. In psycopg 3 you reach it through `cursor.copy()` <https://www.psycopg.org/psycopg3/docs/basic/copy.html>, and the server-side semantics are at <https://www.postgresql.org/docs/16/sql-copy.html>. You `COPY` into the *staging* table; you upsert from staging into the target. Bulk-load the raw, merge the clean.

The eighth thing to internalize is **connection and batch-size discipline**. A connection is expensive to open and is not thread-safe in the way beginners assume; you open one per job (or borrow one from a pool for a long-lived service), wrap your work in a `with` block so it closes even on exception, and let the `with conn:` transaction block commit on clean exit and roll back on exception — psycopg 3's connection and transaction model is documented at <https://www.psycopg.org/psycopg3/docs/basic/transactions.html>. Batch size is the dial between "too many tiny transactions" and "one transaction too big to fit in memory or to roll back cheaply." A batch of 5,000–50,000 rows is the usual sweet spot: large enough that per-transaction overhead is amortized, small enough that a failure rolls back cheaply and progress is visible in the logs. You will tune this empirically in the lab.

The ninth thing to internalize is **structured logging for pipelines**. A pipeline that logs `print("done")` is a pipeline you cannot operate. A pipeline that logs a JSON line per run — `{"run_id": ..., "stage": "load", "rows": 48213, "duration_ms": 1820, "watermark_from": ..., "watermark_to": ...}` — is one you can query, alert on, and debug at 3 AM from a log aggregator without SSHing into anything. Python's standard `logging` module does this with no third-party dependency: a custom `logging.Formatter` that emits JSON, a `run_id` attached via a `LoggerAdapter` or `extra=`, and one log line per stage transition. The HOWTO is at <https://docs.python.org/3/howto/logging.html> and the module reference at <https://docs.python.org/3/library/logging.html>. Every exercise and the mini-project this week emit structured logs, because "rows loaded and how long it took" is the minimum an operator needs and the maximum a `print` will never give you.

By the end of the week you will be the person on your team who can answer "is it safe to re-run the loader?" with "yes, it's idempotent — here's the watermark table, here's the upsert, and here's the test that proves running it five times gives byte-for-byte the same warehouse state" — not with "let me restore from last night's backup and hope."

## Learning objectives

By the end of this week, you will be able to:

- **Distinguish** ETL from ELT and choose between them for a given constraint (transform-before-land vs land-then-transform), and articulate why ELT dominates the cloud lakehouse while ETL still owns the boundary. Cite Kleppmann ch. 10 at <https://dataintensive.net/>.
- **Structure** a batch job as extract → stage → transform → load → publish, using a staging table as the crash-safe landing zone. Cite the PostgreSQL `INSERT` and `TRUNCATE` references at <https://www.postgresql.org/docs/16/sql-insert.html> and <https://www.postgresql.org/docs/16/sql-truncate.html>.
- **Build** an incremental load driven by a stored high-water mark (`max(updated_at)` or `max(id)`), reading only changed rows from the source each run, with a `--full-refresh` fallback. Cite the Kimball ETL techniques at <https://www.kimballgroup.com/data-warehouse-business-intelligence-resources/kimball-techniques/dimensional-modeling-techniques/>.
- **Write** an idempotent upsert with `INSERT ... ON CONFLICT (key) DO UPDATE` (and read the equivalent `MERGE`), so re-loading a row overwrites rather than duplicates. Cite <https://www.postgresql.org/docs/16/sql-insert.html#SQL-ON-CONFLICT> and <https://www.postgresql.org/docs/16/sql-merge.html>.
- **Bulk-load** a CSV into a staging table with psycopg 3's `cursor.copy()`, and explain why `COPY` beats both single-row `INSERT` and one-giant-`INSERT`. Cite <https://www.psycopg.org/psycopg3/docs/basic/copy.html> and <https://www.postgresql.org/docs/16/sql-copy.html>.
- **Advance** a watermark transactionally — in the same transaction as the load it covers — so a crash can never skip a batch, and reason about atomic-batch boundaries for restartability. Cite the psycopg 3 transactions page at <https://www.psycopg.org/psycopg3/docs/basic/transactions.html>.
- **Handle** late-arriving and out-of-order records deliberately: identify when an event-time watermark silently drops a late record, and apply a lookback window, an ingestion/`id` watermark, and a natural-key upsert so the aggregate self-corrects. Cite Kleppmann ch. 11 at <https://dataintensive.net/>.
- **Recognize** the late-arriving-dimension problem (a fact references a dimension member not yet loaded) and apply the inferred-member / placeholder pattern. Cite Kimball at <https://www.kimballgroup.com/data-warehouse-business-intelligence-resources/kimball-techniques/dimensional-modeling-techniques/late-arriving-dimensions/>.
- **Manage** connections with `with` blocks and tune batch size between the two failure modes (too many tiny transactions vs one transaction too large). Cite <https://www.psycopg.org/psycopg3/docs/basic/transactions.html>.
- **Emit** structured JSON logs from a pipeline using the standard `logging` module — a `run_id`, row counts, durations, and watermark transitions per stage. Cite <https://docs.python.org/3/howto/logging.html> and <https://docs.python.org/3/library/logging.html>.
- **Prove** idempotency: run a loader once and five times against the same source and demonstrate, via row counts and a `SUM` checksum, that the warehouse state is identical.

## The "no double-count" promise

C27 treats a single property as non-negotiable for the rest of the course: **a pipeline that loses or double-counts data is broken, no matter how clean the code looks**. From this week forward, every loader you write carries an implicit contract:

```text
Running the loader once and running it five times over the same source
leaves the warehouse in byte-for-byte identical state.
```

A pull request that adds a load path without a test proving that property is, by definition, a pull request that has not been reviewed. The orchestrator in Week 4 will retry your tasks; the streaming job in Week 9 will replay events on recovery — both assume the property holds. The mechanism is not heroics; it is a watermark, an upsert on a real key, and a transaction boundary you drew on purpose. We add a second contract too: **every connection lives in a `with` block, and the watermark advances in the same transaction as the load it covers.** Get those two right and the 3 AM re-run is boring. Boring is the goal.

Structured logging uses the standard-library `logging` module — no third-party dependency. The database is `postgres:16` from Docker Hub. All free, all open source, all GPL-3.0-compatible. Code in this week is written against Python 3.12, psycopg 3.1+, and PostgreSQL 16, and has been kept runnable on exactly that stack.
