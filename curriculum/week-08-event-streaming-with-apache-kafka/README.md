# Week 8 — Event Streaming with Apache Kafka

> *Last week you ran Spark over a pile of files that sat still while you processed them — batch, where the data has a beginning and an end and you read all of it before you answer. This week the data never stops. A clickstream is a river, not a lake; events arrive forever, out of order across the system but in order within a key, and you cannot wait for "all of them" because there is no all. Apache Kafka is the spine of that world: a distributed, replicated, append-only commit log that you write to with producers and read from with consumers, partitioned so it scales and replicated so it survives a broker dying. The single most important sentence this week — say it out loud until it is reflex — is **ordering is only ever per-partition**. Kafka makes no promise about the order of two events in different partitions, ever; it makes an iron promise about two events in the same partition. Everything else — keys, consumer groups, delivery semantics, the schema registry — is downstream of that one fact. By Friday you will have stood up Kafka in Docker, produced a keyed clickstream into a partitioned topic, watched a two-member consumer group rebalance live, and proven that a backward-compatible Avro schema change sails through the registry while an incompatible one is rejected. That topic is not a throwaway: Week 9's Spark Structured Streaming and Flink jobs consume it, so you are building the source the rest of the course streams from.*

Welcome to Week 8 of **C27 · Crunch Data**. Weeks 1 through 7 built the batch world: dimensional models in Postgres, columnar analytics in DuckDB, object storage in MinIO, orchestration in Airflow, transformations in dbt, and last week, distributed processing in Spark. Every one of those assumed a *bounded* dataset — a table, a file, a partition of Parquet — that you read in full and processed to completion. That assumption is about to break. Real systems emit events continuously: a user clicks, a sensor reports, a payment settles, an order ships. There is no moment when the data is "done." This week introduces the substrate that the entire streaming half of the course rests on, and it is not a queue and not a database — it is a *log*.

The mental model to install first is **the log abstraction**. A Kafka topic is an append-only, totally-ordered (within a partition) sequence of records. You only ever append to the end; you never update or delete a record in place. Each record gets a monotonically increasing integer — its *offset* — that names its position in the partition forever. A consumer is just a cursor: it reads from some offset forward, remembers how far it got, and resumes there. This is a profoundly different model from a message queue like RabbitMQ, where a message is delivered once and then gone. In Kafka the record stays on disk until retention expires; ten independent consumers can each read the same record at their own pace, rewind to an old offset, or skip ahead. The log is the source of truth and the consumers are views over it. Jay Kreps's essay "The Log" is the canonical articulation of why this single data structure underpins so much of modern data infrastructure, and the Kafka design docs at <https://kafka.apache.org/documentation/#design> formalize it.

The second thing to install is **partitions and how they relate to ordering**. A topic is split into partitions — independent logs that can live on different brokers and be read in parallel. Partitioning is what lets Kafka scale past one machine's throughput, and it is also the *only* unit of ordering. Within one partition, offsets are strictly increasing and records are read in append order. Across partitions, there is no order at all: partition 0's offset 5 and partition 1's offset 3 have no defined relationship in time. This is not a limitation Kafka could lift with more engineering — it is the price of horizontal scale. If you want two events ordered relative to each other, they must land in the same partition, and you control that with the record's **key**: Kafka hashes the key and maps it to a partition, so all records with the same key (the same `user_id`, say) go to the same partition and are therefore strictly ordered. Pick the key wrong and you either lose the ordering you needed or pile every event onto one partition (a "hot partition") and lose the parallelism you paid for. The lecture's title — *ordering is only ever per-partition* — is the load-bearing idea of the entire week.

The third thing is **consumers, consumer groups, and rebalancing**. A consumer group is a set of consumer processes that cooperatively read a topic, with Kafka guaranteeing that each partition is assigned to *exactly one* member of the group at a time. Five partitions, two consumers: one gets three partitions, the other gets two. Add a third consumer and Kafka *rebalances* — it pauses consumption, recomputes the partition-to-consumer assignment, and resumes, so now each consumer owns one or two partitions. This is how Kafka scales consumption horizontally: more consumers (up to the partition count) means more parallelism, and a consumer crashing triggers a rebalance that hands its partitions to the survivors so no data goes unread. You will *watch* this happen in Lab 08 — start one consumer, start a second, and observe the assignment shift in the logs. The rebalance protocol itself has evolved (eager "stop-the-world" vs. cooperative incremental rebalancing), and which one you use materially affects how much your stream stalls during scaling, covered in Lecture 2.

The fourth thing is **delivery semantics, and what each guarantee actually costs**. There are three: *at-most-once* (a record may be lost but never duplicated — you commit the offset before processing, so a crash mid-process loses that record), *at-least-once* (a record is never lost but may be duplicated — you process then commit, so a crash after processing but before committing reprocesses on restart), and *exactly-once* (no loss, no duplication — the expensive one, requiring the idempotent producer plus transactions). The default and most common is at-least-once, which means **your downstream must tolerate duplicates** — design for idempotency. Exactly-once is real in Kafka (since 0.11, via `enable.idempotence` and transactional read-process-write), but it is not free and not magic: it works within Kafka's own boundaries and demands transactional producers and a careful commit protocol. Knowing which guarantee you have, and what it costs, is the difference between a pipeline that quietly drops payments and one that double-charges customers.

The fifth thing — and the one that separates a stream that lasts from a stream that rots — is the **schema registry**. A Kafka record is just bytes; the producer and consumer have to agree on what those bytes mean. Without a contract, the day a producer adds a field or renames one, every consumer breaks, silently or loudly, and you find out in production. The Confluent Schema Registry stores versioned schemas (Avro, Protobuf, or JSON Schema), assigns each a global ID, and — critically — *enforces a compatibility policy* on every new version: a `BACKWARD`-compatible change (the default) is one a consumer using the new schema can still read with old data; an incompatible change is *rejected at registration time*, before it can poison the topic. The wire format is elegant: a magic byte, the 4-byte schema ID, then the payload — so a consumer reads the ID, fetches the schema from the registry, and deserializes correctly even as schemas evolve. In Lab 08 you will register an Avro schema, evolve it compatibly (adding an optional field — accepted), and try to evolve it incompatibly (removing a required field or changing a type — rejected with a clear error). The registry is the thing that lets a stream survive a year of schema changes.

A note on tooling and alternatives. You will use **Docker** for everything — a single `docker compose` brings up a Kafka broker in **KRaft mode** (no ZooKeeper; KRaft is the post-KIP-500 consensus protocol that lets Kafka manage its own metadata) plus a schema registry. For the Python clients you will use **`confluent-kafka`** (the librdkafka-based client, which is what the schema-registry Avro/Protobuf serializers ship with — prefer it for this week) and we note **`kafka-python`** as the pure-Python alternative for environments where you cannot build native code. And because not everyone wants a JVM on their laptop, the lecture introduces **Redpanda** — a C++ reimplementation of the Kafka protocol with a built-in schema registry and no ZooKeeper/JVM — as a genuine drop-in: the same `bootstrap.servers`, the same client code, a lighter footprint. The challenges give you a Redpanda compose file alongside the Kafka one so you can feel the difference.

By the end of the week you will reason fluently about offsets and consumer groups, choose a partition key that gives you the ordering and the parallelism you actually need, name the delivery guarantee you have and its cost, and use a schema registry to keep a stream evolvable. The mini-project — stand up Kafka, produce a keyed clickstream, run a two-member group through a rebalance, evolve an Avro schema through the registry — is the artifact, and its topic is the input to Week 9. Build it cleanly; you will be consuming it again in seven days.

## Learning objectives

By the end of this week, you will be able to:

- **Explain** Kafka as a distributed, replicated, append-only commit log — not a queue and not a database — and define topic, partition, offset, broker, leader, follower, and in-sync replica (ISR). Cite the Kafka design docs at <https://kafka.apache.org/documentation/#design>.
- **Articulate** why ordering is guaranteed only per-partition, never across partitions, and trace how a record's key determines its partition via the default hash partitioner. Cite <https://kafka.apache.org/documentation/#intro_concepts_and_terms>.
- **Produce** keyed records to a partitioned topic with the `confluent-kafka` Python `Producer`, choosing a key that achieves per-key ordering without creating a hot partition. Cite <https://docs.confluent.io/kafka-clients/python/current/overview.html>.
- **Consume** a topic with a consumer group, and reason about partition assignment, the `group.id`, `auto.offset.reset`, and offset commits. Cite <https://docs.confluent.io/platform/current/clients/consumer.html>.
- **Trigger and read** a consumer-group rebalance: start a second member, observe partitions being reassigned, and explain the difference between eager and cooperative-sticky rebalancing (`partition.assignment.strategy`). Cite <https://kafka.apache.org/documentation/#consumerconfigs_partition.assignment.strategy>.
- **Distinguish** at-most-once, at-least-once, and exactly-once delivery, mapping each to a concrete offset-commit ordering and naming its failure mode (loss vs. duplication). Cite <https://kafka.apache.org/documentation/#semantics>.
- **Configure** producer durability with `acks` (0, 1, `all`) and explain the trade-off between latency and the guarantee against data loss. Cite <https://kafka.apache.org/documentation/#producerconfigs_acks>.
- **Enable** the idempotent producer (`enable.idempotence=true`) and explain how it deduplicates retries within a producer session. Cite <https://kafka.apache.org/documentation/#producerconfigs_enable.idempotence>.
- **Implement** an exactly-once read-process-write loop using Kafka transactions (`transactional.id`, `send_offsets_to_transaction`, `commit_transaction`). Cite <https://docs.confluent.io/platform/current/clients/producer.html#kafka-transactions>.
- **Choose** between time-based and size-based retention and log compaction, and explain when a compacted topic (keep the latest value per key) is the right model versus a retention-windowed topic. Cite <https://kafka.apache.org/documentation/#compaction>.
- **Register** an Avro schema in the Confluent Schema Registry and serialize/deserialize records with `AvroSerializer`/`AvroDeserializer` and `SchemaRegistryClient`. Cite <https://docs.confluent.io/platform/current/schema-registry/index.html>.
- **Predict and verify** schema-compatibility outcomes: apply the `BACKWARD`/`FORWARD`/`FULL`/`NONE` rules to a proposed change and confirm the registry accepts a compatible evolution and rejects an incompatible one. Cite <https://docs.confluent.io/platform/current/schema-registry/fundamentals/avro.html> and <https://docs.confluent.io/platform/current/schema-registry/fundamentals/schema-evolution.html>.
- **Run** Kafka and a schema registry locally in Docker (KRaft mode, no ZooKeeper) and substitute Redpanda as a drop-in alternative with the same client code. Cite <https://kafka.apache.org/documentation/#kraft> and <https://docs.redpanda.com/>.

## Topics covered

- **The log abstraction.** Append-only, immutable records; the offset as a permanent position; the consumer as a cursor, not a recipient; why a log is the source of truth and consumers are views; the contrast with a delete-on-read message queue; replay and rewind.
- **Topics, partitions, offsets.** A topic as a named log split into partitions; partitions as the unit of parallelism *and* of ordering; the offset's monotonic, per-partition, permanent meaning; the high-water mark; the log-end offset; the committed offset vs. the current position.
- **The broker, replication, and ISR.** Brokers as the servers holding partitions; the leader/follower model (one leader per partition handles all reads and writes, followers replicate); the in-sync replica set (ISR); `replication.factor` and `min.insync.replicas`; how `acks=all` interacts with ISR for durability; KRaft (no ZooKeeper) metadata management.
- **Keys and partitioning.** The default hash partitioner (`hash(key) % partitions`); null-key round-robin/sticky behavior; the per-key ordering guarantee; the hot-partition anti-pattern; custom partitioners; what changing the partition count does to existing key→partition mappings.
- **Producers.** The async send model and the delivery callback; `acks` (0/1/all) and the latency-vs-durability trade-off; batching (`linger.ms`, `batch.size`); `retries` and ordering; the idempotent producer (`enable.idempotence`) and the producer ID + sequence number that dedupe retries; `flush()`.
- **Consumers and consumer groups.** `group.id` and the exactly-one-member-per-partition rule; `subscribe` vs. `assign`; `poll` loop mechanics; `auto.offset.reset` (`earliest`/`latest`); offset commit (`enable.auto.commit` vs. manual `commit`); the `__consumer_offsets` topic; consumer lag.
- **Rebalancing.** The group coordinator; what triggers a rebalance (member join/leave, partition count change); eager (stop-the-world) vs. cooperative-sticky incremental rebalancing; `partition.assignment.strategy`; static membership (`group.instance.id`) to avoid rebalances on rolling restarts.
- **Delivery semantics.** At-most-once (commit-before-process, risks loss); at-least-once (process-before-commit, risks duplicates — the common default); exactly-once (idempotent producer + transactional read-process-write); why "exactly-once" is exactly-once *processing within Kafka*, not an end-to-end magic guarantee.
- **Transactions and exactly-once.** `transactional.id`; `init_transactions`/`begin_transaction`/`commit_transaction`/`abort_transaction`; `send_offsets_to_transaction` to atomically commit offsets with output; `isolation.level=read_committed` on the consumer; the read-process-write pattern.
- **Retention and compaction.** Time retention (`retention.ms`) and size retention (`retention.bytes`); the `cleanup.policy` setting; log compaction (keep the latest record per key, tombstones for deletes); when a compacted topic models a changelog/table and when a retention-windowed topic models an event stream.
- **The schema registry.** Why bytes need a contract; subjects and versions; the global schema ID; the wire format (magic byte `0x00` + 4-byte schema ID + serialized payload); `SchemaRegistryClient`; the `TopicNameStrategy` subject naming.
- **Avro, Protobuf, JSON Schema.** Avro's compact binary encoding and reader/writer-schema resolution; Protobuf's field numbers and `.proto` definitions; JSON Schema's verbosity and ubiquity; why Avro is the Kafka-native default and how `confluent-kafka`'s `AvroSerializer` integrates with the registry.
- **Compatibility and evolution.** The compatibility modes — `BACKWARD` (default), `BACKWARD_TRANSITIVE`, `FORWARD`, `FULL`, `NONE`; the concrete rules (adding a field with a default is backward-compatible; removing a required field is not; changing a type is not); what each mode protects and which party (producer or consumer) you upgrade first under each.
- **Redpanda as a drop-in.** A single-binary C++ Kafka-API-compatible broker; no ZooKeeper, no JVM; a built-in schema registry and HTTP proxy; the same client code and config (`bootstrap.servers`); when the lighter footprint matters on a laptop.

## The "ordering is only ever per-partition" promise

C27 treats two things as binding this week. First, you can state — and demonstrate, with a running producer and consumer — exactly what Kafka guarantees about order: nothing across partitions, everything within one, and the key is what puts a sequence of related events in one partition. Second, you can name the delivery guarantee your code has and prove it: at-least-once shows duplicates after a forced restart, exactly-once does not, and you can point at the offset-commit ordering that makes the difference. A pipeline whose author cannot say which guarantee it has is a pipeline that drops or doubles records and nobody knows which.

> **Note on the Week 9 handoff.** The clickstream topic you build in Lab 08 is the input to Week 9's Spark Structured Streaming and Flink jobs. Name it `clicks`, give it a sensible partition count (start with 3), key it by `user_id`, and serialize the value with an Avro schema registered in the registry — Week 9 reads that schema by ID. Sunday's handoff check is: a brand-new consumer group, starting from `auto.offset.reset=earliest`, reads the whole topic and deserializes every record against the registry. If that works, next week starts clean. If it does not, fix it now — debugging a stale topic across a week boundary is the worst place to be.
